# Localization & locale switching (design)

Status: **All four layers implemented** (the compile-time locale switch; a worked
`lib/fr-FR` grammar pack; an `advent` French message + command pack; a forked
French Cloak of Darkness, `sample/cloak_fr.lamp`). The French Cloak is playable
end to end in French — banner, prose, commands, responses, and win banner.
Remaining: a few engine-level English strings and the game-title field — see
Pending.

## What a "language pack" comprises — four layers

Localizing a game touches four distinct things, which switch on different axes:

| Layer | Example | Lives in | Switch behavior |
|---|---|---|---|
| 1. Grammar natives | `the`/`indefinite`, verb agreement, list "and"→"et" | the locale pack (`lib/en-US`) | **Replaced** — a locale pack defines the *same* native names; you load one or the other, never both |
| 2. Library messages | advent's `take_report:"Taken."` → `"Pris."` | the message override registry | **Added** last-wins (see devdocs/messages.md) |
| 3. Command verbs | `"take [taken]"` → `"prendre [taken]"` | a library's `syntax`/`understand` lines | **Replaced/extended** parser grammar |
| 4. Game prose | a game's own room descriptions | the game source | translated by forking the source |

Layer 1 is the engine-facing language data and must be *swapped* (the packs are
mutually exclusive). Layers 2–3 are *additive* overrides keyed by name, loaded
after the thing they translate. Layer 4 is author text, not a message.

> **Layer 1 detail.** A locale pack supplies both the grammar-native *implementations*
> (`the()`, `we()`, `conjugate()`) **and**, since 2026-07-05, the **template-sugar token
> vocabulary** that invokes them — via `sugar` declarations (below). The compiler no longer
> bakes in any language sugar, so a locale can add its own tokens.

## The compile-time locale switch (Part 1 — done)

The locale pack auto-loads right after `lib/sys`, in the slot a swap targets
(load order: `sys → <locale> → <imported libs> → game`). Which pack fills the
slot is chosen at **compile time**, in this precedence:

1. `--locale <tag>` CLI flag (also `--locale=<tag>`), else
2. a `locale "<tag>"` declaration in the game source, else
3. the `en-US` default.

The tag is a directory name under `lib/` (or `<gameDir>/lib/`), e.g. `fr-FR`. A
missing pack is a clear compile error. The flag lets one source compile to many
languages; the declaration lets a game state its own default (a French-authored
game declares `locale "fr-FR"` and needs no flag). The flag wins so a build can
override the declared default.

`locale "<tag>"` is a top-level directive (keyword `locale`, quoted tag — locale
tags carry a hyphen, which is not a valid identifier). It compiles to an inert
`LocaleDecl` AST node with no runtime effect; the compiler reads the tag in a
pre-pass (mirroring `lib` import gathering) to pick the locale dir.

### Per-library locale override files (layers 2–3)

So one switch drives everything, an imported library may ship per-locale override
files: `lib/<library>/locales/<tag>.lamp`. When that locale is active, the file
loads **right after the library's own `.lamp` files** — so its `NAME: "…"`
message overrides and any locale-specific `syntax` lines win last-wins over the
library defaults, while the game file (loaded last overall) can still override
the translation. For the default `en-US` locale no library ships such a file, so
load order and output are unchanged. The `locales/` subdir is not swept by the
normal top-level `.lamp` load; it is pulled in only for the active tag.

## `lib/fr-FR` grammar pack (Part 2 — done)

A worked French locale pack (`lib/fr-FR/{functions.lamp,index.js}`). It defines
every native `lib/en-US` does (same signatures, so any game/world model compiles
against it) with French implementations:

- **Articles** read a grammatical `grammatical_gender` field ("feminine"/"masculine",
  default masculine — a language property of the noun, distinct from the pronoun-set
  `pronouns` field) plus `plural`/`proper`: definite le/la/l'/les (elides to `l'` before
  a vowel or h), indefinite un/une/des (no elision), proper names take no article.
  `an` has no French form and maps to the indefinite.
- **List prose** joins with " et " and no serial comma ("a, b et c"); empty is
  "rien". **Copula** `are(n)` is est/sont. **Numbers** cover the metropolitan
  forms (soixante-dix, quatre-vingts, quatre-vingt-dix, "et un" at 21–61/71).
- **Pronouns** use the same free-text `pronouns` field as en-US: a preset key
  (`il`/`elle`/`iel`/`ils`/`elles`) or a full custom set `subject/object/det/pron/
  reflexive`, read by the `[they]`/`[we]` families — so the nonbinary `iel` and custom
  neopronouns render in French. When `pronouns` is empty the set falls back to
  grammatical gender + number (il/elle/ils/elles), so a game that sets only
  `grammatical_gender` is unchanged. Golden `frpronouns1`.
- **Verbs conjugate in the present tense** (since 2026-07-05 — see "French verb
  conjugation" below): the infinitive citation form `[prendre]` renders the form
  agreeing with the render subject, with polite "vous" for a 2nd-singular viewpoint
  unless `viewpoint_familiar` is set. The English verb vocabulary is also declared and
  passes through unchanged, so a world model's English default templates (`[take]`)
  still parse under this locale. (Participle agreement — "elle est allée" — and other
  tenses are still not modeled; a translation spells those out.)

Tested by the `frlocale1` golden, which selects the pack with a `locale "fr-FR"`
declaration (so the flag-less golden runner exercises it) and checks articles,
elision, plurals, proper names, lists, copula, numbers, ordinals, and case.

## advent French message pack (Part 3 — done)

`lib/advent/locales/fr-FR.lamp` overrides advent's named messages in French
(take/drop/wear/doff/put-on reports + failures + implicit-action parentheticals,
the inventory header/empty/`(worn)`, the `examine_nothing` default, the darkness
line, and the room-contents frame). It loads automatically when fr-FR is active.
Three supporting changes made advent localizable:

- **Non-action prose named** (`lib/advent/rooms.lamp`): the darkness line became
  `darkness_name`/`darkness_description`; the "You see … here." contents frame
  split into `room_contents_intro` + the (local-referencing) contents print +
  `room_contents_outro`, the same split pattern as the inventory `(worn)` marker.
  Both en-US byte-invariant.
- **Inventory article via the locale** (`lib/advent/actions.lamp`): the inventory
  row now renders its article through the locale's `indefinite` instead of
  advent's English-only `with_article`, so it localizes ("une clé"). En-US
  invariant (both produce "a"/"an" for plain items, honoring the `article` enum).
- **A `grammatical_gender` field** on advent's `physical` type (`lib/advent/types.lamp`,
  default "masculine") gives gendered locales a place to read le/la, un/une. It is the
  noun's language property, kept separate from the identity `pronouns` field (which
  drives pronoun rendering in every locale); en-US ignores grammatical gender entirely.
  en-US ignores it; existing fixtures unaffected.
- **Nested-contents parenthetical moved to the locale** (`contained_phrase`): the
  recursive listing's "(in which is a marble)" frame was assembled by string
  concatenation in advent's `render_thing` (with an English `prep_for` helper).
  Prose assembly belongs in the locale, so it became a native
  `contained_phrase(container, inner, count)` — en-US builds the
  preposition/relative/copula clause; fr-FR phrases it with a gender-agreed
  relative pronoun (lequel/laquelle), dans/sur from the supporter flag, and
  "se trouve(nt)". `render_thing` now just calls it; `prep_for` is gone. en-US
  byte-invariant.
- **Supporter listing moved to the locale** (`supporter_phrase`): advent's
  `describe_supporters` built "On the hook is a cloak." in JS (using advent's own
  `display_name`/`with_article`). The sentence became a locale helper —
  en-US "On the hook is a cloak."; fr-FR "Sur l'étagère se trouvent une fiole et
  un crayon." — leaving advent to decide only which supporters and what rests on
  them. This removed the last callers of advent's duplicated
  `display_name`/`with_article`, so both were deleted (the locale now owns all
  naming/articles). en-US byte-invariant.

Tested by the `fradvent1` golden (`locale "fr-FR"` + `lib advent`, an stdin
driver issuing English commands and asserting French responses end to end) and
the `frnested1` golden (nested containers + a scenery supporter, checking the
gendered relative pronoun, dans/sur, elision, and singular/plural agreement
through recursion and a supporter listing).

## French command grammar (layer 3 — done)

`lib/advent/locales/fr-FR.lamp` also supplies the French command grammar:

- **Verb synonyms** via `understand "…" as ACTION` (prendre/poser/porter/enlever/
  examiner/aller/…). This is the same mechanism a game uses for synonyms, so it is
  **additive** — the English templates still match, and under fr-FR the player may
  type either language.
- **Direction words** by reopening advent's `direction` objects to add an
  `understand` synonym list (nord/sud/est/ouest + diagonals + abbreviations); the
  bare `[way]` grammar lets a direction be typed alone.
- A game supplies `understand` synonyms for its **own accented nouns** — the
  parser matches an object by its name/`understand` tokens, not its `printed_name`
  (so `item cle: printed_name "clé"` needs `understand "clé"` to be takeable as
  "prendre clé").

Two general (non-French) fixes this required:

- **UTF-8 stdin** (`src/lamplighter/sandbox/backends/plain.js`): the plain host
  read stdin a byte at a time and decoded each byte as UTF-8 alone, mangling a
  multi-byte character ("clé" → "cl??"). It now accumulates bytes and decodes the
  line once (newline 0x0A never occurs inside a UTF-8 multibyte sequence). ASCII
  byte-invariant.
- **Parser feedback via the message registry**: the engine's "You can't see any
  such thing." / "I don't understand that." now route through `message(name,
  default)` (`parser_cant_see` / `parser_no_understand`), so a translation pack
  overrides them ("Vous ne voyez rien de tel ici." / "Je ne comprends pas
  cela."). The default keeps en-US byte-invariant.

Tested by the `frverbs1` golden (French commands in → French responses, including
the two localized parser failures and an accented noun match).

## French Cloak of Darkness (layer 4 — done)

`sample/cloak_fr.lamp` is the French counterpart of `sample/cloak.lamp`: the
game's own prose (intro, room descriptions, custom action messages) is translated
inline, its objects carry a `grammatical_gender` and `understand` synonyms for
their accented names, and the custom verbs use French syntax (`lire [target]`,
`accrocher [put_item] à [destination]`). It selects French with a `locale "fr-FR"`
declaration; lib/advent supplies the standard French verbs and directions. The
winning path plays fully in French (the `cloak_fr` golden drives it).

To finish the banner, the startup/end prose moved to named messages (en-US
byte-invariant), overridden in the French pack: the title-banner connectives
(`banner_by`/`banner_version`, split around the game-field interpolations), the
end-of-story banner (`story_won`/`story_lost`, in `end_story_rules`), and the
`quit_prompt`.

## Declarable grammar sugar (DONE 2026-07-05)

Template sugar is now **locale-declared**, not compiler-hardcoded. Previously the token
vocabulary — which bare words in `[…]` desugar to which native call — was baked into the
parser (`ARTICLE_SUGAR_FNS`/`PRONOUN_SUGAR_FNS` `const` tables), so a locale could replace a
native but not *add a token*: a language needing sugar the English set lacked (French
`[du]`/`[au]`, German case articles der/dem/den/des, Tamil postpositions) required a compiler
edit. Only `verb` sugar was declarable. That coupling is removed.

**How it works.** A locale pack (or a game) declares its sugar in `.lamp`:

```
sugar operand the            # [the X]  → the(X)        (one reference operand)
sugar operand a as indefinite   # [a X] → indefinite(X)  (word ≠ native)
sugar bare we, us, they, those  # [we] → we(), …          (zero-arg)
sugar bare "we're" as we_re     # apostrophe token → renamed native
```

Two **shapes**: `operand` (`[the X]`, one reference) and `bare` (`[we]`, zero-arg). A token
maps to a native, defaulting to the coerced token; `as NATIVE` renames it when the token
isn't a valid native name (an apostrophe) or differs from the function (`a` → `indefinite`,
so `[a + b]` with a local `a` is left alone). `[We're]`/`[The X]` capitalize as before.

**Mechanism** (mirrors the `verb` path): the prescan collects declarations into a
`Map(token → { native, shape })` (`sugarDecls`), `index.js` unions them across files into
`sugarMap`, threaded to the parser; `desugarSugar`/`desugarOperandSugar` consult it. The
parser ships with **no** hardcoded language sugar. `lib/en-US/functions.lamp` declares the
English set (articles, pronouns, and the D9 contractions); `lib/fr-FR` declares the shared
article/pronoun set (no contractions — French writes those out). Byte-invariant (the
migrated declarations reproduce the old tables); golden `declsugar1` proves a **game** can
add novel tokens (`[yo]`/`[thing X]`) with no compiler change. **Where:**
`src/lantern/{tokenizer,parser_rd,prescan,ast,index}.js`, `lib/en-US`/`lib/fr-FR`
`functions.lamp`. Structural sugar (`[regarding]`, `[is/are LIST]`, `[one of]`, style/marker
words) stays in the parser — it is language-neutral control, not article/pronoun vocabulary.

**Consequence for the Pending items below:** a French pack can now add `sugar operand du`
(with a `du()` native) to stop rephrasing around *du/des*; German case articles become a
locale concern, not a compiler one.

**French alias tokens (2026-07-05):** `lib/fr-FR` declares
`sugar operand le as the, la as the, les as the` and `un/une/des as indefinite`, so the
French message translations in `lib/advent/locales/fr-FR.lamp` read as French end to end
("[Le act.actor] prend [le act.taken]." — swept from the old `[The …]`/`[the …]`). The
alias maps to the same native, which picks the real surface form (gender, number, elision)
from the object — `[le table]` renders "la table", `[un cles]` renders "des clés"; the
token spelling is only what the writer types. Golden `fralias1`.

## Layer 3: locale packs own the default prose (DONE 2026-07-05)

The full mechanism is in devdocs/messages.md ("Default-less references and the
completeness check"). advent's rules now hold no prose — every named message is a
default-less `message NAME` reference; the English text lives in
`lib/advent/locales/en-US.lamp` (loaded under the default locale by the same
per-library mechanism as the French file), the French in `locales/fr-FR.lamp`
(all 105 keys translated), and the compiler errors when the active locale misses a
key — a partial translation fails the build instead of mixing languages.

**Consequence — the fr-FR pack carries no English vocabulary at all.** The English
`verb` list and the English article tokens (`the`/`a`/`an`) are deleted from
`lib/fr-FR/functions.lamp`: no English prose compiles under fr-FR anymore, so
nothing needs them to parse. What remains is purely French (le/la/les, un/une/des,
du/au, the infinitive verb vocabulary) plus the adaptive-pronoun tokens
(`[we]`/`[they]`/`[those]` — the cross-locale API names for the pronoun natives;
French alias spellings for those are still an open naming question). The
`frlocale1` fixture was swept to French tokens accordingly.

**Command grammar too (2026-07-05).** advent's `action` declarations carry no surface
templates anymore: the English `syntax:` blocks (and the direction abbreviations on the
direction objects) moved to `lib/advent/locales/en-US.lamp` as `understand` lines /
direction reopenings, and the French pack's grammar was completed to cover every
player-facing action (open/close, toucher, attaquer, embrasser, pousser/tirer,
donner/montrer with the accented-`à` and bare-`a` template pair, boire/manger, the
session verbs quitter/recommencer/charger/sauvegarder/annuler/score/transcription, the
bare-direction `"[way]"` grammar, and haut/bas/dedans/dehors direction words).
**Behavior change:** a French game no longer accepts English commands — the earlier
"additive, either language" behavior is gone, matching the message split's no-mixing
stance (the French goldens' scripts were swept to French). The IF-universal
abbreviations (x, i, z, q) are declared in the French pack too — conventions, not
English. Exceptions and caveats: debug.lamp's verbs (PURLOIN, SHOWME, …) keep inline
`syntax:` in advent — developer tooling, available under every locale; and unlike
messages there is **no compile-time completeness check for grammar** — a locale that
misses a verb makes it untypeable (still reachable via `try`), with no principled way
to detect it (tracked as an open question in TODO.md).

## French verb conjugation (layer 2 — DONE 2026-07-05)

The foundational fr-FR gap (TODO's "French-contraction strategy", layer 2): `conjugate()`
was identity, so no French text could adapt a verb to its subject — "[Le act.actor] met …"
was hardcoded 3rd-singular and silently wrong for a plural actor. Golden `frconj1`
(vous default, capitalized `[Être]`, 3sg/3pl irregulars and stem-changers, English
passthrough, tu/je/nous). The advent French pack's five `_other` reports now conjugate
("[Le act.actor] [prendre] [le act.taken]." — `[Le …]` notes the actor as subject) —
byte-invariant for singular actors, fixed for plural ones. Its 2nd-person messages
("Vous mettez …") stay spelled out: rewriting them as `[We] [mettre]` would be
byte-invariant under the vous default but wrong under a `je` viewpoint until elision
exists ("je enlève" ≠ "j'enlève"). Design:

- **Citation form is the infinitive** (the French dictionary form): a writer puts
  `[prendre]` in a template and the render agreement (`{person, plural}` from
  `[We]`/`[they]`/`[regarding X]`, defaulting to the viewpoint) picks the surface form —
  "prend", "prennent", "prenez". Capitalization follows the token as usual.
- **Present tense only**, matching en-US. Other tenses, participle agreement
  ("elle est allée"), and negation ("ne … pas" — discontinuous syntax, not a token)
  stay out of scope; a translation spells those out.
- **Politeness (T–V)**: a new language-agnostic narration global
  `viewpoint_familiar` (lib/sys, default false; alongside `viewpoint_person`/
  `viewpoint_plural`). fr-FR renders a 2nd-person-singular agreement as *vous* —
  pronoun set and verb form — unless it is set (then *tu*). Grammatical number is
  untouched (politeness never pluralizes nouns or `[s]`). This also fixes a latent
  inconsistency: `[We]` under fr-FR rendered "Tu" while every French message and the
  parser prose said "vous". en-US ignores the global.
- **Data, not morphology**: `FR_CONJUGATIONS` maps each declared infinitive to its six
  present forms `[je, tu, il, nous, vous, ils]` — explicit entries for irregulars and
  stem-changers (être, avoir, aller, faire, prendre, mettre, voir, boire, ouvrir, tenir,
  sentir, enlever, manger), a helper generating the regular *-er* pattern for the rest
  (poser, laisser, regarder, examiner, embrasser, fermer, porter). A word not in the
  table passes through unchanged — the English verb vocabulary advent's untranslated
  default templates use keeps its identity behavior.
- **Compiler enablement** (two generalizations, both removals of an ASCII-ident
  assumption leaking into locale vocabulary): a `verb` declaration accepts quoted words
  (`verb "être"` — parallel to sugar's `"we're" as we_re`), and the bare-sugar/verb word
  test in the desugarer admits accented letters, so `[être]` routes to
  `conjugate("être")`.

Known limitation: 1st-person-singular elision is not modeled — `[we] [avoir]` under a
`je` viewpoint yields "je ai", not "j'ai". That is the elision-sugar item (`[l' X]`
generalization) also listed under layer 2, tracked separately.

## Pending
- **French `de`/`à` + article contraction — RESOLVED (2026-07-05) via declarable sugar.**
  `the()` can't see a preceding preposition, so "de [the X]" produced the ungrammatical "de le
  caillou". Now fr-FR ships `[du X]`/`[au X]` as its own `sugar operand` tokens (a French-only
  construction English lacks — the exact case declarable sugar was built for). `du()`/`au()`
  contract de/à with the article by gender/number/elision: du/des/de la/de l' and au/aux/à la/à
  l' (golden `frdu1`). A French message writes "… [du X]" instead of rephrasing to dodge "de le";
  the earlier `de_(x)` idea was moot once the token could live in the locale pack. (Full **case
  government** — German genitive with no preposition word — is still open and wants a role marker,
  but that too is now a locale concern, not a compiler one.)
- **Game title — RESOLVED (2026-07-05).** The `game` type's `title` field
  (lib/sys/types.lamp, blank default) already carried the display title — the banner
  prefers it over the identifier when set — but `sample/cloak_fr.lamp` never set it,
  so its banner degraded to "Manteau de l obscurite". It now sets
  `title "Manteau de l'obscurité"` and the banner renders accented and apostrophed.
  Save identity keys on the identifier (`gameInfo().name`), not the title, so setting
  a title never invalidates existing saves.
- **Parser language is now locale-owned (done):** the noun-phrase vocabulary
  (articles, pronouns, self-words) and the two prose renderers — the
  disambiguation prompt ("Which do you mean: … or …?") and the unbound-pronoun
  message (`I don't know what "x" refers to.`) — are installed by the locale via
  the `setParserLanguage` seam (mirroring `setListFormatter`), not hardcoded in the
  runtime. `lib/en-US` installs the English set (byte-invariant); `lib/fr-FR`
  installs French articles (so "le manteau" strips to "manteau"), French pronouns/
  self-words, and French prose ("Lequel voulez-vous dire : …"). This sidesteps the
  interpolable-message problem — the renderer receives the runtime values as
  arguments. Golden `frparser1`.
- **Still engine-level English:** the status line ("Darkness"/"… turns" in
  `startup.lamp`) is host-rendered and not in plain-host output. The session-control
  words recognized by string-compare in `startup.lamp` (`quit`/`q`/`restart`/
  `restore`) are English-keyed — localizing them rides on routing them through the
  grammar path (REVIEW §1.6 / TODO).
